# 常见问题 (FAQ)

本文档解答关于 `cachalot` 的常见疑问。

## 1. 为什么选择 `cachalot`？

在复杂的系统中，我们经常面临“m 个缓存后端（Redis, Memcached, Ristretto, Freecache）”与“n 个缓存增强特性（并发合并, 压缩, 编解码, 逻辑过期, 指标统计）”的组合。
传统的做法往往会导致 **O(m*n)** 的代码爆炸，且难以复用。

`cachalot` 通过解耦 **Store**（存储）、**Factory**（桥接）和 **Decorator**（增强），将复杂度降至 **O(m+n)**。
其核心优势包括：
- **原生泛型支持**：类型安全，无需繁琐的类型转换。
- **高度可定制**：可以自由排列组合增强特性。
- **一致性体验**：无论是单机缓存还是多级缓存，API 体验高度统一。

## 2. 如何选择单级还是多级缓存？

- **单级缓存**：
  - 如果应用是无状态的且对延迟极度敏感，建议使用本地单级缓存（如 `ristretto`）。
  - 如果数据量巨大且需要多实例共享，建议使用远端单级缓存（如 `redis`）。
- **多级缓存**：
  - 典型场景：L1 (本地) + L2 (远端)。
  - **优势**：兼顾了本地缓存的极速响应和远端缓存的大容量/共享性。
  - **权衡**：增加了系统复杂度和弱一致性风险（L1 与 L2 之间可能存在短暂不一致）。

## 3. 逻辑过期与物理过期的区别？

- **物理过期 (Physical TTL)**：
  - 数据在存储层（如 Redis）过期后被直接删除。
  - **缺点**：高并发下，缓存失效的一瞬间，大量请求可能穿透到数据库（缓存击穿）。
- **逻辑过期 (Logical TTL)**：
  - 数据物理上长期保存（或 TTL 远大于逻辑 TTL），但在数据结构中记录一个过期时间戳。
  - **优势**：当检测到逻辑过期时，`cachalot` 会利用 `stale-while-revalidate` 模式，立即返回旧数据，并在后台异步更新缓存。这彻底消除了高并发下的毛刺。

## 4. 为什么要有“字节路径” (Byte Path)？

在许多缓存库中，序列化是固定在最外层的。这意味着如果你想对数据进行压缩，流程通常是：`Object -> Marshal -> Compress -> Store`。
`cachalot` 引入了 **Byte Path** 概念，允许在 `factory` 层面决定数据是否进入“字节处理阶段”。

通过 `WithCodec` 和 `WithCompression` 等 API，开发者可以：
- 在同一链路上透明处理复杂的压缩算法（如 Zstd, Gzip）。
- 避免在不需要字节化（如本地 interface 缓存）的场景下引入多余的开销。
- 允许在字节流层面对数据进行统一的加密或签名，而无需侵入业务代码。
